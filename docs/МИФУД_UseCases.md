## UC-01: Регистрация

**Акторы:**
- Студент (основной актор)
- Система

**Предусловия:**
- Пользователь не авторизован
- Открыта страница регистрации

**Основной поток:**
1. **Студент** вводит имя, email и пароль
2. **Студент** нажимает «Зарегистрироваться»
3. **Система** проверяет корректность введённых данных
4. **Система** проверяет уникальность email
5. **Система** создаёт запись в таблице users
6. **Система** перенаправляет студента на страницу входа

**Расширения:**

_3а. Поля заполнены некорректно_
- 3а.1 **Система** определяет пустые или невалидные поля
- 3а.2 **Система** выделяет ошибочные поля и показывает сообщение
- 3а.3 Возврат к шагу 1

_4а. Email уже зарегистрирован_
- 4а.1 **Система** определяет, что email уже существует в таблице users
- 4а.2 **Система** уведомляет студента о необходимости использовать другой email
- 4а.3 Возврат к шагу 1

**Постусловия:**
- В таблице users создана новая запись
- Студент может войти в систему

**Предусловия:**
- Студент авторизован

## UC-06: Оформление заказа

**Акторы:**
- Студент (основной актор)
- Система

**Предусловия:**
- Студент авторизован
- В корзине есть хотя бы одно блюдо
- Открыта страница оформления заказа для пользователя

**Основной поток:**
1. **Студент** открывает страницу оформления заказа для пользователя
2. **Система** отображает состав корзины и итоговую сумму
3. **Студент** указывает время получения
4. **Студент** опционально добавляет комментарий
5. **Студент** нажимает «Оформить заказ»
6. **Система** проверяет количество активных заказов студента
7. **Система** создаёт запись в таблице заказов со статусом в обработке
8. **Система** создаёт записи в таблице заказанных товаров для каждого блюда
9. **Система** вычисляет итоговую сумму
10. **Система** перенаправляет студента на страницу статуса

**Расширения:**

_3а. Время получения не указано_
- 3а.1 **Система** определяет пустое поле времени получения
- 3а.2 **Система** показывает ошибку валидации
- 3а.3 Возврат к шагу 3

_6а. Достигнут лимит активных заказов_
- 6а.1 **Система** определяет, что у студента уже 3 активных заказа
- 6а.2 **Система** уведомляет студента о невозможности оформить новый заказ
- 6а.3 Оформление не происходит

_10а. Студент поставил галочку «Забронировать стол»_
- 10а.1 **Система** перенаправляет студента на страницу выбора стола (UC-07)

**Постусловия:**
- В таблице заказов создана новая запись со статусом в обработке
- В таблице заказанных товаров созданы записи для каждого блюда
- Корзина очищена

## UC-07: Бронирование стола

**Акторы:**
- Студент (основной актор)
- Система

**Предусловия:**
- Заказ успешно оформлен (UC-06)
- Студент поставил галочку «Забронировать стол»
- Открыта страница выбора столов

**Основной поток:**
1. **Система** отображает список столов с количеством мест
2. **Система** проверяет доступность каждого стола на время получения заказа
3. **Студент** выбирает стол
4. **Система** проверяет отсутствие конфликта по времени
5. **Система** создаёт запись в таблице броней, привязанную к ID заказа
6. **Система** перенаправляет студента на страницу статуса заказа

**Расширения:**

_2а. Все столы заняты на выбранное время_
- 2а.1 **Система** определяет, что нет доступных столов
- 2а.2 **Система** уведомляет студента
- 2а.3 Студент может отказаться от брони и перейти к странице статуса

_4а. Стол занят в момент подтверждения_
- 4а.1 **Система** обнаруживает конфликт по времени
- 4а.2 **Система** уведомляет студента и предлагает выбрать другой стол
- 4а.3 Возврат к шагу 3

**Постусловия:**
- В таблице броней создана запись, привязанная к заказу
- Информация о броне отображается на странице статуса

## UC-08: Просмотр статуса заказа

- Студент (основной актор)
- Система

**Предусловия:**
- Студент авторизован
- У студента есть хотя бы один заказ

- 2а.1 **Система** отображает сообщение об отсутствии заказов
- 2а.2 **Система** предлагает перейти в меню

**Постусловия:**
- Студент видит актуальный статус своих заказов

## UC-09: Отмена заказа

**Акторы:**
- Студент (основной актор)
- Система

**Предусловия:**
- Студент авторизован
- Заказ имеет статус в обработке или подтвержден

**Основной поток:**
1. **Студент** нажимает «Отменить» на странице статуса
2. **Система** меняет статус заказа на отменен
3. **Система** проверяет наличие брони, привязанной к заказу
4. **Система** меняет статус брони на отменен (если бронь была)
5. **Система** обновляет отображение страницы статуса

**Расширения:**

_1а. Заказ имеет статус ready или completed_
- 1а.1 **Система** блокирует кнопку отмены
- 1а.2 Отмена невозможна

**Постусловия:**
- Статус заказа изменён на отменен
- Связанная бронь (если была) также отменена

## UC-13: Смена статуса заказа

**Акторы:**
- Сотрудник столовой (основной актор)
- Система

**Предусловия:**
- Сотрудник авторизован
- Открыта страница заказов для сотрудников

**Основной поток:**
1. **Сотрудник** открывает страницу управления заказами

**Расширения:**

_4а. Переход статуса недопустим_
- 4а.1 **Система** блокирует недопустимые переходы
- 4а.2 Статус не изменяется

**Постусловия:**
- Статус заказа обновлён в таблице заказов

## UC-15: Автоматический подсчёт суммы заказа

**Акторы:**
- Система (основной актор)

**Предусловия:**
- Студент подтвердил оформление заказа (UC-06, шаг 5)
- Записи в таблице заказанных товаров созданы

**Основной поток:**
1. **Система** вызывает метод calculate_total() для нового заказа
2. **Система** получает все позиции из заказанных товаров по ID блюда
3. **Система** суммирует цена × количество для каждой позиции
4. **Система** записывает результат в поле итоговая стоимость таблицы заказов

**Расширения:**

_2а. Позиции заказа отсутствуют_
- 2а.1 **Система** записывает итоговую стоимость, равную 0

**Постусловия:**
- Поле истоговая стоимость в таблице заказов заполнено корректным значением
- Итоговая сумма отображается студенту на странице статуса
